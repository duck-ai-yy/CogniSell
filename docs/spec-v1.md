# spec-v1 — 销售第二大脑 CRM

> 维护者：鸭鸭（项目负责人）。subagent 不改本文件。
> 本 spec 只含产品逻辑：不含技术选型、不含 UI 视觉设计、不含竞品分析（那些在 plan / decisions.md）。

## niche（差异化坐标）

给**长周期工业销售**的「认知可审计」第二大脑。不同于 Clay / Attio / Salesforce Einstein 把记忆塞进向量库，这里每条关系都是人类可读、可审计、带认知状态的图谱边。

## 用户与场景

用户 = 工业自动化部件供应商的销售代表。三个场景是「认知同步」发生的三个时刻：

1. **录入**（名片）：会展上拍一摞名片 → 识别成可勾选联系人 → 语音选谁留、谁深挖。
2. **更新**（cold mail）：爬潜客信息 → 多 agent 抗辩出接近策略 → GenUI 邮件，发送前确认（改动被记住）→ 往来邮件提炼进图谱。
3. **重同步**（catch-up）：某客户长期没联系 → 系统主动提醒 → 结合图谱近况 + 上次接触给出 catch-up 建议。

## 产品内核（地基，不可退让）

- 客户关系 = 知识图谱；图谱跟着销售一起进化；人机认知保持**双向**同步。
- 「确认→记住」回路的真正定位 = **认知同步协议**（不只是偏好记忆）。
- co-evolution 靠时间：图谱记得自己怎么变（`supersedes`），可叙述认知变迁史。

## 认知状态（边的生命周期）

`proposed`（机器提议）→ 人确认 → `confirmed`；人纠正 → `corrected`；过期/被取代 → `retired`（记 `supersedes`）。

## 无摩擦同步原则

- 高置信 + 低风险 → 静默 auto-confirm，不打扰人。
- 低置信 **或** 即将拿它对外行动（发信、落库承诺）→ 进决策收件箱，人语音/单击秒清。
- 阈值随用户清卡变松（越用越无摩擦）。

## 异常分支表

| 异常 | 用户应看到什么 |
|---|---|
| OCR / 抽取失败 | 明确提示"这张没认出来，要不要手动补"，不静默丢弃 |
| GLiNER2 模型不可用 | fallback 到 mock 抽取，并提示"当前用离线兜底" |
| 发信失败（mock 也模拟失败态） | 提示失败原因，邮件留在草稿不丢 |
| 图谱里查无此人/此公司 | catch-up 给"信息不足，建议先补背调"的诚实回应，不编造 |
